### Лекційна частина

**Як працює сучасний веб (за 5 хвилин)**
Коли ти вводиш URL у браузері:
1. DNS-сервер перетворює "lovable.dev" у IP-адресу (типу 76.76.21.21)
2. Браузер надсилає HTTP-запит на цей IP
3. Сервер (десь у хмарі) повертає HTML, CSS, JavaScript
4. Браузер рендерить це у візуальну сторінку

Це базовий цикл. Все інше — оптимізації цього процесу: CDN (швидші сервери ближче до користувача), кешування (не питати те саме двічі), bundling (мінімізація файлів). AI-білдери ховають всю цю кухню, але розуміти її варто, бо інакше не зрозумієш, чому "сайт повільний" або "не індексується Google".

**Стек сучасного веб-сайту: HTML / CSS / JS**
- *HTML* — структура (заголовки, параграфи, кнопки)
- *CSS* — стилі (кольори, розміри, верстка)
- *JavaScript* — інтерактивність (форми, анімації, API)

Усі сучасні сайти — це варіація цих трьох. Frameworks (React, Vue, Svelte, Next.js) — це абстракції над JavaScript, які дозволяють писати компоненти і повторно використовувати код. AI-білдери генерують стек React/Next.js + Tailwind CSS — це поточний стандарт індустрії.

**SPA, SSR, SSG — що це і чому має значення**
- *SPA (Single Page Application)* — весь сайт це одна HTML-сторінка, JavaScript керує що показувати. Швидкі переходи, але повільне перше завантаження, погано для SEO.
- *SSR (Server-Side Rendering)* — кожна сторінка генерується на сервері при запиті. Добре для SEO, повільніше якщо багато трафіку.
- *SSG (Static Site Generation)* — сторінки збираються заздалегідь, віддаються як файли. Найшвидше, найкраще для блогів, лендінгів.

Lovable, Claude Code зазвичай генерують SSR/SSG через Next.js — оптимально для більшості сайтів.

**Спектр AI-інструментів для веб у 2026**

*Drag-and-drop (Wix, Squarespace, Webflow):* як 5 років тому. Готові блоки, перетягуй. Швидко, але обмежено стилем шаблонів.

*AI website builders (Lovable, v0.dev від Vercel, Bolt.new, Wix AI):* описуєш у чаті, отримуєш робочий сайт. Найшвидше "0 to 1". Кастомізація обмежена тим, що AI розуміє.

*AI coding agents у браузері (Jules, Replit Agent):* AI пише код у хмарі, ти бачиш результат у preview. Більше контролю, ближче до коду.

*AI coding agents локально (Claude Code, Cursor, Windsurf):* AI працює з твоєю файловою системою, повний контроль. Найгнучкіше, але потребує термінала і базового розуміння файлової структури.

*Класична розробка з AI-помічниками (VS Code + Copilot/Cursor):* для досвідчених. AI як співавтор, не як автор.

Правило вибору: швидкість важливіша за контроль → Lovable. Контроль важливіший за швидкість → Claude Code. Не хочеш термінал → Jules або Lovable.

**Performance: чому це не дрібниця**
Дослідження Google: якщо сайт вантажиться довше 3 секунд — 53% мобільних користувачів закриває. Кожна додаткова секунда — мінус 7% конверсії.

Core Web Vitals — три метрики, за якими Google оцінює сайти і впливає на SEO:
- *LCP (Largest Contentful Paint):* час до того, як з'явилось найбільше зображення на екрані. Має бути <2.5s.
- *INP (Interaction to Next Paint):* як швидко сайт реагує на клік. Має бути <200ms.
- *CLS (Cumulative Layout Shift):* як сильно "стрибає" контент під час завантаження. Має бути <0.1.

Lighthouse у Chrome DevTools покаже всі три. 80+ за всіма — нормально, 95+ — добре.

**Accessibility — це не про мораль, це про досяжність**
15% людей у світі мають якусь форму інвалідності. Якщо твій сайт не accessible — ти відрізаєш цей сегмент. Плюс:
- Незрячі користувачі (screen reader)
- Літні люди (важче читати дрібний текст)
- Люди з тимчасовими обмеженнями (зламана рука, спрацював на сонці)
- SEO-боти, які "читають" сайт як screen reader

WCAG 2.2 — стандарт accessibility. Базові правила:
- Контраст тексту і фону мінімум 4.5:1 (перевір на webaim.org/resources/contrastchecker)
- Alt-текст на всіх зображеннях (опис того, що зображено)
- Заголовки в правильній ієрархії (H1 → H2 → H3, не пропускай)
- Усе має керуватись з клавіатури (без миші)
- Видимий фокус на елементах форм

Lighthouse Accessibility audit показує більшість проблем. Ціль: 95+.

**SEO crash course**
Google ранжує сторінки за сотнями факторів, але є базовий мінімум:
- *Title tag (60 символів):* головне ключове слово на початку
- *Meta description (155 символів):* опис, який покажеться у пошуковій видачі
- *H1, H2, H3:* семантична структура з ключовими словами
- *Alt-тексти:* для зображень
- *URL-структура:* читабельна, з ключовими словами (`/coffee-shop-uzhhorod`, не `/p?id=234`)
- *Швидкість і mobile-friendly:* фактори ранжування
- *OG-tags:* як виглядає посилання, коли його шерять у соцмережах

Перевір на: metatags.io (preview meta-тегів), search.google.com/search-console (як Google бачить твій сайт).

**Mobile-first як філософія**
50%+ трафіку інтернету — мобільний. Не "адаптувати десктоп під мобільний", а навпаки: спочатку дизайнити для маленького екрана, потім розширювати на десктоп. Це змінює рішення:
- Великі тач-цілі (кнопки мінімум 44×44px)
- Короткі тексти, скан-friendly
- Один CTA на екран
- Швидке завантаження

AI-білдери часто роблять "desktop-first з мобільною версією як зменшеним десктопом". Це працює погано. Завжди перевіряй на справжньому телефоні.

3. Generate Report → чекай ~30 сек

**Найкращі практики**
- **Замінюй placeholder контент відразу.** AI-білдери вставляють текст-заглушки ("Lorem ipsum", "John Doe says..."). Не заміниш — забудеш, потрапить у production.
- **Тестуй на справжньому телефоні, не тільки в Chrome DevTools.** Реальний рендеринг часто інший.
- **Lighthouse-аудит обов'язковий.** Чекай 80+ за всіма метриками.
- **Не оптимізуй передчасно.** Перший раз — щоб працювало. Другий — щоб виглядало добре. Третій — оптимізація. У зворотньому порядку нічого не вийде.
- **Git commit після кожного робочого стану.** Особливо коли працюєш через Claude Code — він може щось зламати, треба буде відкочуватись.
- **OG-image — це твоя обкладинка.** 80% людей побачать твій сайт через шеринг у соцмережах. OG-image — це те, що вони побачать перед тим, як клікнути. Витрать 15 хв на гарну.
- **Швидкість > краса.** Анімований hero з 5MB відео = -30% конверсії. Прибирай "круте", якщо воно повільне.
- **Один CTA на головному екрані.** Якщо у hero 3 кнопки — користувач не знатиме, на яку тиснути, і не натисне жодну.

### Пастки і поради
- Перш ніж писати промпт у Lovable — мати готову карту сайту: які блоки, в якій послідовності, що в кожному.
- Не намагатись з першого промпта отримати "ідеальний" сайт. Робота йде ітераціями: "змени це", "додай те", "колір трохи темніший".
- Зображення на сайт — генерувати окремо (Day 2 інструменти), потім завантажувати. AI-білдери самі підставляють зображення-плейсхолдери, які треба замінити.
- Тестувати на справжньому телефоні, не тільки в Chrome DevTools.

### Debrief
1. Lovable vs Claude Code/Jules — який інструмент дав більше відчуття контролю? Який дав швидше робочий результат? Це були різні відповіді?
2. Де AI-сайт зламався: мобільна версія, контент, accessibility, дизайн? Що довелось правити руками і скільки часу це зайняло?
3. Як ти зараз формулюєш правило "який тип сайту для якого інструменту"? Дай 2–3 чітких сценарії.
4. Що ти НЕ зміг отримати від AI скільки б не пробував? Це обмеження інструмента чи твоїх промптів?
5. Чи зробив би ти зараз сайт для реального клієнта через ці інструменти? За скільки грошей і за скільки часу? Обґрунтуй.
